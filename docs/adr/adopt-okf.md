# Adopt the Open Knowledge Format

## Status

Accepted

## Context

Arkouda invented its own frontmatter schema — `id`, `title`, `abstract`, `status`, `date` — when the CLI was first built (see [basic-adr-cli.md](basic-adr-cli.md)). The schema was never the point; it was the minimum needed to make a directory of Markdown files machine-readable.

Since then the [Open Knowledge Format][okf] (OKF) has specified almost exactly that: a directory of Markdown files with YAML frontmatter, designed to be authored by people, generated by agents, and consumed by both. OKF calls the directory a *knowledge bundle* and each document a *concept*. Its stated goals — readable without tooling, parseable without bespoke SDKs, diffable in version control, portable across tools and organizations — are the goals arkouda already had.

Two things follow from a bespoke schema that OKF removes:

- **An agent that understands OKF still cannot read an arkouda ADR.** It looks for `type`, finds nothing, and has to special-case the format. The whole value of arkouda's schema being strict is lost if nothing else speaks it.
- **The `id` field is redundant.** OKF defines a concept's id as its path within the bundle minus the `.md` suffix. Arkouda's validator already required the filename stem to equal `id` (`E005`), which is a check that only exists because the same fact is stored twice.

Arkouda's ADR-specific metadata — `status`, `deciders`, `superseded_by` — has no OKF equivalent, but OKF §4.1 explicitly permits producer-defined keys and requires consumers to tolerate them. Nothing needs to be dropped.

OKF is at v0.1 and arkouda is pre-1.0. This is the cheapest moment to move.

## Decision

Treat every configured ADR directory as an OKF v0.1 knowledge bundle.

**Frontmatter** maps onto OKF's vocabulary:

| Before      | After         | Notes                                                     |
| ----------- | ------------- | --------------------------------------------------------- |
| —           | `type`        | Required by OKF. Always `Architecture Decision Record`.    |
| `id`        | *removed*     | The concept id is the bundle-relative path without `.md`.  |
| `title`     | `title`       | Unchanged.                                                 |
| `abstract`  | `description` | OKF's name for a one-sentence summary.                     |
| `date`      | `timestamp`   | ISO 8601 date or datetime.                                 |
| `tags`      | `tags`        | Unchanged.                                                 |
| `status`    | `status`      | ADR extension, retained.                                   |
| `deciders`  | `deciders`    | ADR extension, retained.                                   |
| `superseded_by` | `superseded_by` | ADR extension, retained.                                |

**Structure** follows the spec:

- `index.md` and `log.md` are reserved (OKF §3.1) and are never treated as concepts.
- Discovery recurses into subdirectories, so `security/mtls.md` is the concept `security/mtls`.
- `arkouda index` generates the bundle-root `index.md` (OKF §6), grouping concepts under status headings for progressive disclosure. It declares `okf_version: "0.1"` — the one place OKF §11 permits frontmatter in an index. `arkouda new` refreshes an index that already exists; it never creates one, because OKF §9 makes indexes optional.

**Validation** stacks two layers. OKF conformance (§9) is the floor: the document parses and declares a non-empty `type`. Arkouda's ADR contract sits on top: `type` must be `Architecture Decision Record`, `status` must come from the controlled list, `timestamp` must be ISO 8601, and the Nygard body sections must be present.

OKF's permissive-consumption rule (§9) decides what is a warning rather than an error. A bundle is never rejected for declaring an OKF version arkouda does not implement (`E013`) or for having a stale `index.md` (`E014`). Both are reported; neither fails CI.

No backward compatibility is offered. Arkouda is pre-1.0, the old schema has no external consumers, and supporting both would mean two code paths through the validator forever.

## Consequences

### Positive

- An arkouda ADR bundle is now readable by any OKF consumer without special-casing, and arkouda can be pointed at any OKF bundle to at least parse it.
- One less redundant field, and two fewer validation rules (`id`-is-a-slug and `stem`-matches-`id` collapse into a single check on the concept id).
- Recursive discovery means monorepos can nest ADRs by area inside one bundle, not just list sibling directories in `.arkoudarc.toml`.
- `index.md` gives an agent a single file to read to see every decision and its one-line summary, without opening any of them.

### Negative

- Every existing ADR must be migrated. There is no compatibility shim; `arkouda check` fails loudly on the old schema, which is the intended way to find them.
- `--abstract` becomes `--description` and `--sort date` becomes `--sort timestamp`. Scripts and agent skills that used the old names break.
- `type` is a constant on every ADR — pure ceremony from arkouda's point of view. It buys interoperability, which is the whole trade.

### Neutral

- `log.md` (OKF §7) is reserved and validated when present, but arkouda does not generate it. Git history already records what a log file would.
- Arkouda's bodies keep `#` for the ADR title and `##` for Nygard's sections, so citations sit under `## Citations` rather than OKF's conventional `# Citations`. The spec's heading conventions assume documents with no title heading; the ADR template does not.

## Alternatives Considered

### Keep the bespoke schema

Zero work, and the validator stays as it is. But arkouda's pitch is that ADRs should be machine-readable for agents — and a format only one tool can read is machine-readable in the weakest possible sense.

### Adopt OKF but keep `id` as an extension

OKF tolerates unknown keys, so `id` could have stayed. This would have made the migration a pure addition of `type`. It also would have preserved exactly the redundancy OKF was designed to remove, and kept a validation rule (`E005`) whose only job was to police that redundancy.

### Support both schemas behind a flag

Every read path would branch, the validator would need two rule sets, and `arkouda new` would need a default. The cost is permanent; the benefit lasts until the last old ADR is migrated. Arkouda is pre-1.0 and its ADR collections are small.
